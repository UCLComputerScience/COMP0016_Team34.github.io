Testing

We testing the system in various ways to ensure system stability and robustness

Manual Testing

Manual testing was used throughout the development of the system in order to identify both visible and hidden defects in system. At each stage of development (caller ui design, server side data reception, receptionist ui design etc.) we made sure tested using temporary variables to model the system as if it were really being used.

Django Test

The Django was tested using cURL for example,

curl -X POST https://team34-comp0016-2020.azurewebsites.net/login/ -F "username=ReceptionistTest" -F "password=APassword" -c cookies.txt
curl -X POST https://team34-comp0016-2020.azurewebsites.net/addURLID/ -F "id= 137504652159298206822370043863588881692" -F "url=google.com" -F "description=google" -b cookies.txt
curl -X GET https://team34-comp0016-2020.azurewebsites.net/loginTest/ -b cookies.txt

These commands were used to send data to the server to ensure the format is suitable for the server - this means it doesn't matter how data is sent by the client. The html returned can be captured and due to Djangos debug mode any errors were reported Similarly issues found when performing user tests, with ourselves, clients etc was another valuable way to find bugs and then fix them.

Java server/client

The interaction between the receptionsit client and the server is achieved using Java http methods. The client sends 'GET' and 'POST' requests to the server. This interaction was tested by sending information to the server and receiving the response code (200 if successful and an error code if unsuccessful)

Client testing

We asked our clients to also test the system during a live demo so they could see first-hand what the system was capable of, as well as being able to receive feedback on the system features. This was useful to identify what features they still wanted, but also to see how the system was perceived by people outside of the development team, which is important.
